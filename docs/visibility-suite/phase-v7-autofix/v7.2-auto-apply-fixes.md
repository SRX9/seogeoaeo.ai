# V7.2 — Auto-apply fixes

- **Status:** ☐ Not started
- **Type:** ⚙️ Deterministic application
- **Pillar:** 🔵 / 🟣 / 🟢 · Catalog (cross-cutting)
- **Depends on:** findings with `fix_payload` from V1.1, V1.3, V3.3, V6.5
- **Unlocks:** the "audit → fix in one app" loop

## Goal

One-click apply stored `fix_payload`s — insert JSON-LD, write `llms.txt`, patch robots.txt, swap in
optimized answer blocks — to content we control, or as copy-paste / downloadable snippets / PR for
external sites.

## Files to create

- `src/lib/visibility/apply-fix.ts` — `applyFix(findingId): ApplyResult`.
- `src/app/api/visibility/fix/route.ts`.
- UI "Apply fix" / "Copy snippet" actions on findings.

## Build steps

1. Each `fix_payload` declares a `kind` (`jsonld` | `llms_txt` | `robots_txt` | `answer_block` |
   `meta`) + the content + a target.
2. **For content we can reach — apply directly.** That's drafts *and* anything published
   through the publishing connectors (WordPress, Ghost, Dev.to, …): patch the meta/schema/
   answer block and **push the update back through the same connector**. This is the widest
   commercially-true version of "she fixes it on your actual site" — treat the connector
   path as first-class, not a nice-to-have. Store the before-state so every apply is
   revertible (V8.5 depends on this).
3. **For surfaces we can't reach:** produce a copy-paste snippet, a downloadable file
   (`llms.txt`), or a ready PR/patch — we can't edit their server.
4. **Verify the lift:** after applying, re-run the relevant analyzer (V2.1/V3.2/V1.1) and confirm the
   score moved; record before/after on the finding (`is_resolved`).
5. Test: applying a generated schema fix marks the finding resolved + raises the schema score.

## 📐 Build from (source of truth)

No new algorithm — consumes the `fix_payload`s produced by V1.1 (robots.txt), V1.3 (llms.txt), V3.3
(JSON-LD), V6.5 (answer blocks). Re-scores with the same modules to verify.
