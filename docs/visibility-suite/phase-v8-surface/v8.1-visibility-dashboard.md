# V8.1 — Visibility dashboard page

- **Status:** ☐ Not started
- **Type:** ⚙️ UI/plumbing (no LLM)
- **Depends on:** [V2.3](../phase-v2-scoring-spine/v2.3-unified-audit.md) (composite score),
  [V8.4](v8.4-credit-metering.md) (audit spend)
- **Unlocks:** V8.2 (fix queue lives under this nav section), V6.3 (trend data accumulates here)

## Goal

Ship `/visibility` — the hero surface: one 0–100 score, six sub-scores, score trend, last-audit
summary, and a "Run audit" button. Add a compact Visibility score card to `/dashboard`. This is
the page the whole product is judged by: a site owner must understand it in five seconds.

Two rules from `01-product-surface.md` are binding here: **owner language** (sub-scores render
their display names — "Quotable answers", "Site health" — from the display map, grouped under
the three surface pillars, never raw "Citability"/"E-E-A-T"/"Schema") and **score context**
(the number is never shown alone — always with the delta vs the previous audit *and* an
industry baseline: "You: 68 · typical for SaaS sites: 74". `audits.business_type` already
exists; the baseline is a median over prior audits of the same type, falling back to the
global median under a minimum sample size).

## Files to create

- `src/app/(app)/visibility/page.tsx` — shell + sections.
- `src/lib/visibility/display.ts` — the display-name map (internal key → owner label + one-line
  explainer) for pillars and sub-scores; the single source for these strings.
- `src/components/visibility/score-hero.tsx` — big gauge, band label (Excellent…Critical), delta
  vs previous audit, industry-baseline line.
- `src/components/visibility/subscores.tsx` — six sub-score tiles (owner labels from
  `display.ts`), grouped under the three surface pillars, each with a "what's this" tooltip.
- `src/components/visibility/audit-runner.tsx` — "Run audit" button: shows credit cost from
  `CREDIT_COSTS.visibility_audit`, confirms, starts the Workflow, polls status.
- `src/components/visibility/score-trend.tsx` — line chart over past audits (renders empty-state
  until ≥2 audits).
- `src/app/api/visibility/summary/route.ts` — latest audit + sub-scores + previous score.
- `src/app/api/visibility/audits/route.ts` — POST start audit (meter first), GET list/status.
- `src/components/dashboard/visibility-card.tsx` — compact score card for `/dashboard`.

## Build steps

1. Add `Visibility` to `primaryNav` in `src/components/layout/app-shell.tsx` (after Overview).
2. Follow the section-loading convention: page shell renders instantly; score hero, sub-scores,
   and trend each load via their own `<Section>` + skeleton from granular endpoints.
3. `POST /api/visibility/audits`: spend `visibility_audit` credits (V8.4 helper, refId = audit
   id) **before** starting the V0.3 Workflow; return 402-style out-of-credits payload consistent
   with article generation.
4. Empty state (no audits yet): explain the score in one sentence, show the button. First-run
   experience matters more than the populated one.
5. While an audit runs: show progress from Workflow status (pages fetched / analyzers done);
   audit runs also append to Activity.
6. Dashboard card: current score + band + delta arrow, links to `/visibility`.
