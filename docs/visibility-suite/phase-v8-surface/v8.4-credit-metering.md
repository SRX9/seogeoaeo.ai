# V8.4 — Credit metering for visibility jobs

- **Status:** ☐ Not started
- **Type:** ⚙️ Plumbing (no LLM)
- **Depends on:** [V0.3](../phase-v0-foundation/v0.3-data-model-orchestration.md) (audit
  workflow to meter); existing billing stack
- **Unlocks:** V8.1 (audit spend), V8.3 (tool-run spend), V6.2/V6.4 (report/benchmark spend),
  V8.5 (plan-gated monitoring cadence)

## Goal

Extend the existing credit system to every visibility job. Users buy **outcomes** (an audit, a
fix, a report, a tool run) — never "tool access." Reuse, don't rebuild: `CREDIT_COSTS` in
`src/lib/billing/credits.ts` is the single source of truth; `credit_ledger` provides the audit
trail and `refId` idempotency exactly as article generation does today.

## Files to touch

- `src/lib/billing/credits.ts` — add keys (draft values from `01-product-surface.md`, tune
  before launch): `visibility_audit: 50`, `answer_run: 10`, `competitor_benchmark: 40`,
  `pdf_report: 25`, `tool_run_basic: 5`, `tool_run_ai: 20`. **No per-fix key** — auto-fixes
  are plan-included with a monthly cap (see `01-product-surface.md` pricing: "an employee is
  salaried, not a taxi meter").
- `src/lib/usage/credits.ts` (or wherever the spend helper lives) — a
  `spendForVisibilityJob(action, refId)` wrapper if the existing helper doesn't already
  generalize; follow the article-generation call pattern exactly.
- `src/lib/billing/plans.ts` — plan features from the `01-product-surface.md` plans table:
  monitoring cadence, tracked-prompt count + answer-check cadence (V5.5), **auto-fix monthly
  cap**, competitors benchmarked, PDF-report inclusion — expressed as plan caps, mirroring how
  weekly article caps work today.
- Pricing page + billing UI — surface the new costs (UI reads `CREDIT_COSTS`, so mostly
  copy updates).

## Build steps

1. Add the cost keys. Keep values proportionate to existing anchors (article 100, research 20):
   an audit at 50 must feel obviously cheaper than an article; a basic tool run at 5 must feel
   nearly free.
2. Spend **before** work starts, `refId` = run/audit id, so retries can't double-charge
   (ledger's partial unique index already enforces this).
3. Out-of-credits path: same behavior as article generation, including the existing
   out-of-credits email flow (Claudia persona) — extend its copy to mention visibility jobs.
4. Free surfaces stay free and unmetered: the public quick snapshot and free tools (V1.5,
   V8.6) are rate-limited and cached by domain, never touch credits, and must not require
   auth. **Proof is also never metered** — traffic panels (V6.6) and viewing answer-share
   history (V5.5) cost nothing; only *running* an on-demand answer check spends `answer_run`.
5. Scheduled agent audits and answer checks (V8.5/V7.3/V5.5) spend from the monthly allowance
   at the plan's cadence; if the workspace is out of credits the agent skips, logs to
   Activity, and the out-of-credits email covers it. Auto-fixes decrement the plan's monthly
   fix cap, not credits; the first full audit for a new workspace is covered by the V8.6
   `first_audit_free` grant (recorded in the ledger).
6. Tests: idempotent double-spend attempt, out-of-credits rejection, ledger entries carry the
   right `reason`/`refType`.

## Acceptance

- [ ] All credit-priced visibility actions declared in `CREDIT_COSTS`; no hardcoded costs
      anywhere else; no per-fix charging exists anywhere.
- [ ] Spends are idempotent by refId and appear in the ledger with correct reason/refType.
- [ ] Public snapshot + free tools spend nothing and work logged-out; proof surfaces are
      never metered.
- [ ] Plan tiers gate monitoring cadence, tracked prompts, and the auto-fix cap like they
      gate weekly article caps today.
- [ ] Out-of-credits behavior (API response + email) matches article generation.
