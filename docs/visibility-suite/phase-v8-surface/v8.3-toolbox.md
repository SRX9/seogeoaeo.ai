# V8.3 — Toolbox (standalone tool pages)

- **Status:** ☐ Not started
- **Type:** ⚙️ UI/plumbing (no LLM logic of its own)
- **Depends on:** [V8.4](v8.4-credit-metering.md); enough shipped analyzers to feel alive —
  build after Phase V3 (V1.1–V1.4, V2.1–V2.2, V3.1–V3.3 ≈ 9 tools)
- **Unlocks:** per-run monetization of every subsequent analyzer for free

## Goal

Ship `/tools` — every analyzer as its own standalone page for technical users, priced per run
in credits. One registry + one shared shell so each new tool costs one registry entry and one
result renderer, **never** a bespoke page. Toolbox runs call the exact same engine modules as
the audit (engine contract #2) and their findings land in the same fix queue.

## Files to create

- `src/lib/visibility/toolbox-registry.ts` — `TOOLBOX: ToolboxEntry[]` where each entry declares:
  `slug`, `name`, `pillar`, `description` (one owner-language sentence), `inputKind`
  (`domain` | `url` | `page-or-text`), `costKey` (`tool_run_basic` | `tool_run_ai`),
  `run(input) → ToolResult` (the analyzer's dual-mode entry-point), `ResultView` component ref.
- `src/app/(app)/tools/page.tsx` — grid grouped by pillar (SEO / AEO / GEO / Cross-cutting),
  each card: name, description, cost chip.
- `src/app/(app)/tools/[slug]/page.tsx` — shared `ToolRunner` shell: input form by `inputKind`,
  cost + confirm, run, render `ResultView`, past runs for this tool.
- `src/components/visibility/tool-runner.tsx` + `src/components/visibility/tool-results/*.tsx`
  (one small renderer per tool; many can share a generic findings-list renderer).
- `src/app/api/tools/[slug]/route.ts` — POST: meter (refId = run id) → `run()` → persist.
- `src/lib/db/schema/visibility.ts` — add `tool_runs` table: `id`, `workspace_id`, `brand_id`,
  `slug`, `input` (jsonb), `result` (jsonb), `score`, `created_at`. Findings still go to
  `audit_findings` (with a nullable `tool_run_id` ref) so the fix queue sees them.

## Build steps

1. Define `ToolboxEntry` + registry with the tools shipped so far; wire `/tools` grid from it.
2. `ToolRunner` shell: input → cost confirmation (read from `CREDIT_COSTS`, show balance) →
   POST → skeleton → result. Deterministic tools should feel instant; LLM tools show progress.
3. Persist runs; findings insert through the same repository as audits (dedupe per V8.2).
4. Every result view ends with two links: "See all issues in your Fix queue" and "Let Claudia
   handle all of this → Visibility". The Toolbox always points back to the core loop.
5. Sidebar: add `Toolbox` as a secondary/lower nav item — it must not compete with Visibility
   in the hierarchy, and it is never part of onboarding.
6. As later phases land (V4, V5, V6.5), adding each tool to the Toolbox = registry entry +
   renderer. Add that step to those tickets' definition-of-done from now on.
